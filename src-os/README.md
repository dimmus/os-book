# src-os (phase 0 tutorial tree)

This is a standalone tutorial milestone:

- builds a minimal freestanding kernel (`HELLO INIT`)
- embeds/loads it from a standalone UEFI application
- calls `ExitBootServices`
- jumps to the kernel entry
- prints to COM1 (serial)
- Stage 3: PIC + PIT + IRQ0 (see `OS-6-irq-pic-study.md` in the book tree)
- Stage 4: Local APIC timer + MMIO map (see `OS-7-lapic-study.md`)
- Stage 5: #PF handler + `CR2` probe (see `OS-8-page-fault-study.md`)
- runs under QEMU with OVMF

## Run

From `src-os/`:

```sh
make run
```

If your system OVMF path differs, set:

```sh
make run OVMF_FD=/path/to/OVMF.fd
```

## How `uefi/kernel_blob.inc` is produced

The UEFI app embeds the kernel as a generated C array.

Build pipeline:

1. `kernel/Makefile` builds `build/kernel.elf` from `kernel/init.cpp`, `idt_entry.S`, `irq_entry.S`, `lapic_entry.S`, and `pf_entry.S`.
2. `kernel/Makefile` converts ELF -> raw binary:
   - `llvm-objcopy -O binary build/kernel.elf build/kernel.raw`
   - `python3 kernel/pad_kernel_raw.py build/kernel.elf build/kernel.raw` pads to the first `PT_LOAD` `MemSiz` so `.bss` (e.g. the IDT array) is present in the blob (`llvm-objcopy` alone often stops at `FileSiz`).
3. `uefi/Makefile` runs:
   - `python3 gen_kernel_blob.py <kernel.raw> <kernel_blob.inc>`
4. `uefi/gen_kernel_blob.py` reads `build/kernel.raw` and writes `uefi/kernel_blob.inc`
   with:
   - `kernel_blob[]` (byte array)
   - `kernel_blob_len` (size in bytes)

So the source for `gen_kernel_blob.py` is always the raw file generated by the
kernel build step: `build/kernel.raw`.

## Troubleshooting: Stage 1 OK but no Stage 2 lines

1. Rebuild **kernel then UEFI** from `src-os/`: `make clean && make build` (not only `make -C kernel`).
2. The UEFI app embeds `kernel.raw` via `uefi/kernel_blob.inc`. If `efi_main.obj` was newer than `kernel_blob.inc`, an incremental build could leave a **stale blob** in `BOOTX64.EFI`; the `uefi/Makefile` now ties `efi_main.obj` to `kernel_blob.inc` so that cannot happen.
3. If you see `STAGE 1: paging done` but not even `STAGE 2: begin`, the CPU is still running an **old** embedded kernel.
4. The kernel is **linked at VMA 0** but loaded at `kernelPhys`. Never dereference `&g_idt` as a bare link address; use `kernelPhys + offset` (see `idtTable()` in `kernel/init.cpp`).
5. **`STAGE 2: begin` then hang**: previously caused by clearing the IDT at an unmapped link address, or by mapping the firmware GDT exhausting the page-table scratch pool — the tree now uses a **boot GDT on the stack** plus `lretq` to `CS=0x08`.
6. **`#BP dispatcher` but not `done`**: `iretq` reloads **SS** from the interrupt frame; if it still holds UEFI’s selector (e.g. `0x30`) while your boot GDT only has `0x08`/`0x10`, **`iretq` faults** — the ISR patches the saved SS before `iretq` (`kernel/idt_entry.S`).
